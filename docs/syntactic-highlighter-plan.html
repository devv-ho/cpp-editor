feat: Regex 기반 Syntactic Highlighting 레이어 추가

feat active 2026-05-17 branch: feat/lsp-e2e-tests PR #15

개요

clangd의 semantic tokens는 일반적인 C++ 파일에서 약 86개의 토큰만 커버합니다 — 함수, 타입, 변수, 매크로, 연산자처럼 의미론적(semantic) 정보가 있는 식별자들만 해당됩니다. int, const, return 같은 C++ keyword, #include 같은 전처리기 지시문, 주석, 문자열 리터럴은 전혀 내려주지 않습니다. 결과적으로 파일의 상당 부분이 default 색상으로 표출됩니다.

이 계획은 state machine 기반 syntactic highlighting 레이어를 먼저 실행해 모든 공백을 채우고, LSP semantic 레이어가 겹치는 구간을 덮어쓰는 2단계 파이프라인을 도입합니다. 외부 의존성 없이 VS Code 수준의 highlighting 품질을 달성합니다.

LSP "operator" → Yellow (기존 유지). Syntactic 연산자는 "syn_operator" → Cyan을 사용해 충돌을 방지하며, LSP가 매칭하는 구간은 LSP가 덮어씁니다.

State Machine 설계

파일 전체를 왼쪽에서 오른쪽으로 한 번만 스캔합니다. 6개 상태를 가지며, 현재 상태에 따라 같은 문자열을 다르게 해석합니다. 이 방식이 "// not a comment", /* "not a string" */ 같은 케이스를 정확하게 처리하는 핵심입니다.

STRING '\n' → "string" span [start, EOL) 생성 (미종료) · CODE 복귀 (다음 줄은 정상)
CHAR '\\' → 다음 문자 skip
CHAR "'" → "string" span [start, pos+1) 생성 · CODE 복귀
CHAR '\n' → "string" span [start, EOL) 생성 (미종료) · CODE 복귀

PREPROC 그 외 + '\n' → "preprocessor" span [start, EOL) 생성 · CODE 복귀

Merge 알고리즘 (LSP 우선)

핵심 기술 결정

Regex-per-category 대신 State machine
카테고리별 regex를 독립적으로 적용하면 "// not a comment", /* "not a string" */ 같은 케이스를 올바르게 처리할 수 없습니다. 단일 왼→오 패스의 state machine은 lookahead·backtracking 없이 이를 정확히 해결합니다.
파일 전체 스캔, 라인 단위 출력
여러 줄에 걸친 블록 주석을 처리하려면 state가 줄을 넘어 유지되어야 합니다. 파일 전체를 한 번 스캔하고, 출력은 기존 token_map과 동일한 per-line SpanList map으로 만듭니다.
LSP wins on overlap
LSP는 의미론적으로 더 정확한 정보를 가집니다 (예: string이 타입인지 keyword인지). 겹치는 구간에서는 LSP span이 항상 우선합니다.
"syn_operator" 별도 타입 문자열
LSP는 "operator" → Yellow를 내려줍니다. Syntactic 레이어는 "syn_operator" → Cyan을 사용해 충돌을 방지하며, LSP가 매칭하는 구간은 LSP가 덮어씁니다.
Header-only SyntaxHighlighter.hpp (src/drivers/)
FTXUI include 없이 순수 로직만 포함해 unit test에서 독립적으로 사용 가능합니다. 새로운 CMake 타겟·의존성 없이 기존 GLOB_RECURSE에 자동으로 포함됩니다.
미종료 리터럴 복구
닫는 "/' 없이 줄 끝을 만나면 해당 라인을 string span으로 처리하고 다음 줄부터 CODE 상태로 복귀합니다. 컴파일 에러에 해당하는 상황을 시각적으로 올바르게 표현합니다.

시스템 전체 영향

성능
render_buffer()가 렌더 프레임마다 O(n) 스캔을 한 번 추가 실행합니다. ~40줄 파일에서는 무시 가능하며, 대용량 파일에서 지연이 느껴질 경우에만 프로파일링합니다.
오류 전파
highlight_file은 빈 입력에 빈 map을 반환하며 예외를 던지지 않습니다. render_buffer는 syntactic map이 비어도 LSP-only highlighting으로 gracefully fallback합니다.
스레드 안전성
highlight_file은 공유 상태 없는 순수 함수입니다. mutex·locking 불필요.
변경되지 않는 불변식
colorize_line 시그니처, cursor 주입 로직, 기존 LSP overlay 렌더링 (diagnostics, hover, completion, locations, symbols, inlay hints) 전부 변경 없음.
API 표면
colorize_line·token_color는 anonymous namespace에 존재. 외부 호출자 없음.
통합 커버리지
U3의 merge unit test가 live renderer·FTXUI screen 없이 syntactic + LSP 상호작용을 검증합니다.

리스크 & 완화 방안

리스크완화 방안
State machine이 \\ escape를 잘못 처리해 라인 나머지가 잘못된 색상으로 표출 Unit test: "escape: \"", '\\' 케이스; STRING/CHAR skip 로직에 한정된 수정
여러 줄에 걸친 블록 주석이 O(n²) span-list 증가를 유발 라인마다 별도 span entry를 가지므로 비용은 O(주석 줄 수); 이차 성장 없음
Merge 로직이 LSP span 대신 syntactic span을 살리는 우선순위 역전 버그 U3 merge unit test가 LSP span 생존·syntactic span trim을 명시적으로 단언
대용량 파일에서 렌더 지연이 눈에 띄게 증가 단일 O(n) 패스; 사용자가 지연을 보고할 때만 프로파일링
highlight_file이 \r\n 줄 끝으로 호출됨 CODE 상태에서 \r를 공백으로 취급; \n을 항상 라인 경계로 사용
